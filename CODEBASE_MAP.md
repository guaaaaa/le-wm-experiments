## At a glance

This repo is the research code for **LeWM** — a world model that learns, from videos + actions of
a robot playing, to predict *what its camera will see next* — except it never predicts pixels: it
predicts a compact 192-number summary (a "latent") of the next frame. It is deliberately tiny
(~15–18M params): `jepa.py` + `module.py` define the whole model, `train.py` the whole objective.
Everything else — environments, datasets, the planner that uses the model to control a robot —
lives in two installed libraries (`stable_worldmodel`, `stable_pretraining`). We use it for two
studies: the completed Push-T planning characterization, and the in-flight Memory-Maze
extended-context memory study.

## Stack

- **Language**: Python 3.10, PyTorch 2.12 (cu130, Blackwell), Lightning, Hydra configs
- **Heavy-lifting deps**: `stable_worldmodel` (swm: envs, HDF5 data, CEM planner, World harness),
  `stable_pretraining` (spt: training Module/Manager, ViT backbone, transforms)
- **Data**: HDF5 on `/mnt/nas3/lewm/memory-maze/` (Memory Maze) and `/mnt/nas2/lewm/datasets/`
  (Push-T); `STABLEWM_HOME` env var is the root for datasets/checkpoints
- **Hardware**: 4× shared Blackwell GPUs (empty-GPU rule), MUJOCO_GL=egl for rendering

## Mental model

Think of three boxes and two clocks.

**The three boxes:** (1) an **image encoder** (ViT-tiny) that turns ONE frame into one 192-D
latent — it has no concept of time; (2) an **action encoder** (a small MLP) that turns a block of
raw actions into one 192-D conditioning vector; (3) a **predictor** (6-layer causal transformer) —
the ONLY place time exists — that looks at up to `history_size` past frame-latents and, steered by
the action embeddings via AdaLN, emits the *next* latent. Training teaches box 3 to predict box 1's
output one step ahead (teacher-forced, never multi-step), with a single regularizer (SIGReg)
stopping the trivial "make every latent identical" cheat — no EMA teacher, no stop-gradient.

**The two clocks:** one *model step* = `frameskip` env frames (Push-T: 5, Memory Maze: 4). The
model only ever sees every `frameskip`-th image, and one prediction jumps that far. The raw actions
in between are not discarded — they're stacked into one "action block" vector (5×2=10-D for
Push-T, 4×6=24-D one-hots for Memory Maze), so control is per-frame but prediction is per-block.

**Planning** (inference only) runs the predictor as an imagination engine: CEM proposes 300
candidate action sequences, the model rolls each forward in latent space, and the candidate whose
*final* latent lands closest to the encoded goal image wins. That endpoint-only cost is the single
most consequential design fact in the repo — it's why Push-T planning only works executed
open-loop (k=H), a finding documented in `outputs/pusht/RESULTS.md`.

## Guided reading order

Read in this order; each session builds on the previous. Times assume careful reading.

**Session 1 — the model (60–90 min): `module.py` → `jepa.py`**
1. `module.py:189-214` `Embedder` — smallest piece; note Conv1d(k=1) = per-block linear, no time mixing.
2. `module.py:56-85` `Attention` — standard causal SDPA; note NO length assumption except…
3. `module.py:244-285` `ARPredictor` — …the 3-row (now `history_size`-row) `pos_embedding` at :262,
   the ONLY sequence-length-bound parameter in the whole model.
4. `module.py:88-111` `ConditionalBlock` + `modulate()` (:6) — THE non-obvious part: actions never
   become tokens; `adaLN_modulation` turns the action embedding into shift/scale/gate that
   modulate the frame token at the same time index. Zero-init = blocks start as identity.
5. `jepa.py:29-55` `encode`/`predict` — watch the `(b t)` fold at :36: time folded into batch =
   proof the ViT is time-blind. CLS only at :38 (patch tokens discarded).
6. `jepa.py:61-110` `rollout` — inference-only autoregression: window slides (`emb[:, -HS:]`),
   model feeds on its own predictions. Compare with training (next session) — this asymmetry
   (teacher-forced train vs self-fed inference) is why drift compounds with horizon.
7. `jepa.py:112-153` `criterion`/`get_cost` — the endpoint-only MSE cost (`[..., -1:, :]` at :121).
   ✅ Checkpoint: you should now be able to answer "why does context cap at history_size?" and
   "where exactly do actions influence the prediction?"

**Session 2 — training (45–60 min): `train.py` + `config/train/*` + `utils.py`**
1. `config/train/lewm.yaml` then `data/pusht.yaml`, `data/memory_maze.yaml`, `model/lewm.yaml` —
   note `num_steps = num_preds + history_size` (window W = context C + 1 target) and
   `num_frames: ${history_size}` (the pos-table auto-sizes).
2. `train.py:17-45` `lejepa_forward` — the entire objective in 28 lines. Key indexing:
   `ctx = emb[:, :C]`, `tgt = emb[:, num_preds:]`; causal masking makes one window supervise
   context lengths 1…C simultaneously. tgt is NOT detached (no stop-grad!).
3. `module.py:10-36` `SIGReg` — read with `../findings.txt` §7 next to it: project batch latents
   on 1024 random directions, force each 1-D shadow to look Gaussian (Epps–Pulley test) — this
   alone prevents collapse. Input must be (T,B,D), hence the `.transpose(0,1)` at train.py:40.
4. `train.py:47-140` `run()` — dataset load, the normalizer loop (:61-70, note the
   `normalize_action` gate we added for one-hots), `input_dim = frameskip × action_dim` (:70),
   window-level 90/10 split (:76-79 — leaky, see Hot spots), spt.Module wiring, SaveCkptCallback.
5. `utils.py` — 60 lines: img preprocessor (ImageNet norm), picklable ZScoreNormalizer (no eps
   clamp!), checkpoint callback.
   ✅ Checkpoint: explain why pred_loss alone would collapse, and what exactly one gradient step
   supervises for a W=17 Memory-Maze window (answer: 16 one-step predictions, contexts 1–16).

**Session 3 — the data pipeline (45 min): installed swm, read in `.venv/`**
1. `stable_worldmodel/data/formats/hdf5.py:40-135` `HDF5Dataset` — the h5 schema (`ep_len`,
   `ep_offset`, flat columns), `_load_slice`: observations strided by frameskip (:121-122),
   `action` kept dense, images HWC→CHW (:129).
2. `stable_worldmodel/data/dataset.py:40-75` — clip indexing (`span = num_steps×frameskip`, one
   clip per valid start) and the action reshape `(W·fs, A) → (W, fs·A)` (:71) that builds blocks.
3. `experiments/memory_maze/convert_npz_to_h5.py` + its tests — our converter: schema, Zstd
   chunking, and the **one-step action shift** (npz `action[t]` leads INTO obs t; LeWM wants the
   block DEPARTING obs t; verified empirically, corr +0.978).
   ✅ Checkpoint: given episode length 1001 and W=17/fs=4, how many clips? (1001−68+1=934) and
   which action rows feed model step t? (dense rows 4t…4t+3, post-shift.)

**Session 4 — planning & evaluation (60 min)**
1. `eval.py` — the whole eval protocol: sample (episode, start) pairs (:120-145), init env to
   `state[start]`, goal = `state[start+25]`, planner sees only the goal IMAGE, success = state

2. `stable_worldmodel/envs/pusht/env.py:314-355` — `step()` (PD control substeps) and
   `eval_state` (:347): success = ‖goal[:4]−cur[:4]‖<20px AND angle<20° — note it includes the

3. `stable_worldmodel/policy.py:294-431` `WorldModelPolicy` — PlanConfig (H/k/action_block), the

   when the buffer drains (k blocks). This is where "open-loop vs receding-horizon" physically lives.
4. `stable_worldmodel/solver/cem.py:31-230` `CEMSolver` — 300 samples × 30 iters × 30 elites over
   a diagonal Gaussian in (H × block-dim) space; seeded `torch_gen`.
5. `stable_worldmodel/world/world.py:353-490` `World._run_iter`/`evaluate` — batched envs, alive

## Entry points

### `train.py` — training
- **Runs**: `python train.py data=memory_maze img_size=64 model.encoder.patch_size=8 history_size=16 normalize_action=false`
- **Flow**: Hydra compose → `swm.data.load_dataset` (HDF5, windowed clips) → transforms (ImageNet
  norm + z-score) → `hydra.utils.instantiate(cfg.model)` → spt.Module(`lejepa_forward`) →
  Lightning fit → `SaveCkptCallback` writes `$STABLEWM_HOME/checkpoints/<name>/weights_epoch_N.pt`.

### `eval.py` — planning evaluation
- **Runs**: `python eval.py --config-name=pusht.yaml policy=pusht/lewm` (+ our overrides:
  `model_device`, `allow_long_horizon`, `output_dir`, results.json dump)
- **Flow**: load pickled JEPA → sample N (episode,start) pairs seeded by `cfg.seed` → World.evaluate
  (CEM plans toward goal image, env terminates on state-distance success) → success_rate + videos.

### `experiments/pusht_horizon/run_planning.py` — sweep driver
- Expands condition grids (T0/T1/T1o/T2…), spawns `eval.py` subprocesses, aggregates results.json.

### `experiments/memory_maze/convert_npz_to_h5.py` — dataset converter
- npz episodes → swm-schema h5 (Zstd, action-shifted). CLI or `convert()`.

### `convert_checkpoint.py` — HF weights → pickled `_object.ckpt` (ViT key remap included).

## Data flow

### Training step (Memory Maze config, B=128, C=16, fs=4)
1. Dataset yields `pixels (128,17,3,64,64)` (strided frames) + `action (128,17,24)` (dense,
   reshaped, pre-shifted at conversion) — `dataset.py:67-72`
2. `lejepa_forward` (train.py:17): nan_to_num(actions) → `model.encode` → `emb (128,17,192)`,
   `act_emb (128,17,192)`
3. One predictor call on `[:, :16]` → `preds (128,16,192)`; targets `emb[:, 1:]`
4. `loss = MSE(preds, tgt) + 0.09·SIGReg(emb.T)` → Lightning backward/step (AdamW 5e-5, bf16)

### Planning step (Push-T eval)
1. Policy buffer empty → CEM asks `model.get_cost(info, candidates (1,300,H,10))`
2. `rollout`: encode current frame (1 real latent), autoregress H steps with sliding
   `history_size` window → final latents (300,)
3. `criterion`: MSE(final latent, goal latent) → CEM refits Gaussian on top-30, 30 iterations
4. Winning plan unstacked to H×5 per-frame actions → executed k×5 env steps before replanning
   (k=H ⇒ fully open-loop; the paper's protocol)

## Key abstractions

| abstraction | file | one-liner |
|---|---|---|
| `JEPA` | `jepa.py` | the model: encoder+projector, action_encoder, predictor+pred_proj; `history_size` attr governs rollout window |
| `ARPredictor`/`ConditionalBlock` | `module.py` | causal transformer; AdaLN-zero action conditioning; pos-table = the context cap |
| `Embedder` | `module.py` | action block → 192-D; first layer = slot-wise embedding tables in disguise |
| `SIGReg` | `module.py` | the anti-collapse loss; input (T,B,D), reduces over batch |
| `HDF5Dataset`/`Dataset` | swm `data/` | flat h5 + ep_len/ep_offset; windowing, frameskip striding, block reshape |
| `World`/`EnvPool` | swm `world/` | batched envs, alive-masking, evaluate protocol |
| `WorldModelPolicy`/`PlanConfig` | swm `policy.py` | H (plan) vs k (execute) vs action_block (frameskip); the action buffer |
| `CEMSolver` | swm `solver/cem.py` | 300×30×30 diagonal-Gaussian search, seeded |
| `spt.Module`/`Manager` | spt | Lightning wrapper that binds `lejepa_forward` + optimizers |

## Conventions and patterns

- Hydra everywhere: any config key overridable on the CLI; experiments pass overrides, never edit
  yaml in place.
- `EXPERIMENT_RULES.md` is binding: contract-first code (CONTRACTS.md → blind test author →
  implementer), outputs layout, run naming, shared-machine limits (empty-GPU rule, NAS storage).
- Shape notation in docstrings: `b`=batch, `t`=time, `s`=CEM samples, `d`=embed dim.
- Long jobs = idempotent scripts + `driver.log` + `final_status.txt` on the data volume, launched
  detached; agents/sessions read state from disk, never from memory.

## Hot spots and risks

- `jepa.py:rollout` **history_size coupling** — was hardcoded 3; now defaults to
  `self.history_size`. Any NEW consumer must not pass a literal.
- `train.py` **normalizer loop** — z-scores every non-pixel column; for one-hot actions this
  destroys the encoding and NaNs unused indices (no eps clamp in `utils.py:22`). Gated by
  `normalize_action:false` for Memory Maze; Push-T keeps legacy behavior.
- `train.py:74-79` **window-level train/val split** — overlapping windows leak across the split;
  val loss is optimistic. For honest numbers use the separate eval h5 (episode-level split).
- swm `envs/pusht/env.py` — success includes the **agent's** position (not just the T);
  `success_threshold`/`coverage_arr` are dead code; the rendered green goal is `env.goal_pose`
  (default), NOT the eval goal — videos must set `goal_pose` manually (see
  `experiments/pusht_horizon/make_precision_videos.py`).
- **Planning nondeterminism**: identical seeds give ±few-% success (multithreaded float
  reduction); report mean±CI, never single runs.
- **Memory-Maze npz action convention** differs from Push-T's (leading-into vs departing; zero
  placeholder at row 0 vs NaN at row end) — handled once in the converter; do not "fix" it again
  downstream.
- `pusht.yaml` `num_steps` arithmetic: `num_steps = num_preds + history_size` must hold or the
  loss tensors silently misalign (only broadcasting saves/betrays you).

## Things that are NOT obvious

1. **The ViT never sees time; the predictor never sees pixels.** All temporal reasoning happens
   over 192-D latents in a 6-layer transformer whose sequence is ≤ history_size long.
2. **Actions are not tokens.** They modulate LayerNorms (AdaLN-zero). Attention only ever attends
   over frame latents.
3. **Training is single-step teacher-forced; inference is multi-step self-fed.** The model is
   never trained on its own predictions — that asymmetry is the root cause of the measured
   open-loop drift and the H≥10 planning collapse.
4. **The planning cost is endpoint-only** (`criterion` scores just the last rolled latent), which
   makes receding-horizon execution (k<H) collapse and open-loop (k=H) mandatory.
5. **One causal window trains every shorter context for free** — a C=16 model contains trained
   C=1…16 behaviors; that's the lever the memory study's context sweep stands on.
6. **frameskip ≠ action repeat.** All intermediate raw actions are kept, stacked into the block.
7. `interpolate_pos_encoding=True` covers the ViT's *spatial* positions only; the predictor's
   *temporal* table never interpolates — it's the hard context cap.

## Open questions

- Does the checkpointed `*_object.ckpt` pickle survive our `jepa.py` edits long-term? (Back-compat
  `getattr` handles history_size, but future edits could break unpickling — prefer weights+config.)
- Is spt's `random_split` window leak intentional upstream (fast val proxy) or an oversight worth
  reporting?
- CEM `var_scale=1.0` in normalized action space — is the init variance well-matched to Memory
  Maze's one-hot blocks, or will the first iterations waste samples far off the simplex?
- Why does the paper's Table 5 report training-seed variance but not eval-set variance? (Our
  500-episode validation suggests the latter dominates: 96.0 seed-42 vs 89.0 pooled.)
